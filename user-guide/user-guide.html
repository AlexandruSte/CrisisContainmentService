1. Introduction

Crisis Center (Crisis Containment Service) is a Web platform which allows managing situations of crisis (for instance, earthquakes, floods, fire, tsunamis) by authorities or different organisations.

The application integrates Google Crisis Map and services like Google Person Finder, in order for the system automatically provide information used for finding a missing person and regarding eventual shelters and saving routes in cases of emergency.

The website will provide users with the ability to check again the past events in their area and if they were solved. Last but not least, users will be presented a variety of articles regarding how to help organisations, what to do in a crisis, first-aid, a page where world-wide current events are being shown and also a page where other organisations in partnership with Crisis Center can be found.

2. Creating an account and logging in

In order to have access to recent news in your local area, you need to create an account. Don't worry, your informations are safely stored, as we respect GDPR procedures.

2.1 How to Create an Account

  1. On the home page, press the button "Sign Up" in the upper-right corner of the page. You will be redirected to the Register page.
  2. On the Register page, you will have to choose if you want to create an account as a citizen or an authority. All of the information in the form is required, as it is needed to be able to recognize emergencies in your local area. Complete the fields carefully, choosing a valid email (if we need to contact you or send notifications to you) and a strong passphrase.
  3. After you have completed filling out the fields, press the Submit! button. This will take you back to the home page. From now on, you will be able to log in.
Important: If you have signed as an authority, you will be asked to validate the information you have provided via email. (future functionality)

2.2 How to Log in

  1. In order to login, you have to press the button Login located in the upper right section of every page.
  2. You need to provide the email you registered with and your passphrase.
  3. After that, you can press the button "Submit". If you provided the correct information, then the login is successful and you will be redirected to the login page.

2.3 I Forgot My Password

Do not worry if you forgot your password! You can always reset it by pressing the "Forgot password" button located in the login form. Then, a mail will be sent to you and will take you to further steps of recovering it. (future functionality)

3. What Our Website Offers

Through Crisis Center, we want to provide anyone that enters our website with the essential information regarding how to help somebody and what is happening around the world.

This is why we provide users with breaking news and articles. You do not need to log in or have an account in order to access this information.

3.1 Breaking News

The Home Page is where breaking news titles are being shown. In case you want more information on what happened, you can enter the News page, where you can find short texts that will lead you to other safe websites where full articles are being presented.

3.2 Worldwide Emergencies and Current Events

You can see critical areas around the globe on the News Page, where Google Crisis Map is integrated. All the data presented is verified and always being updated.

3.3 See Partners

We have associated with diverse organizations which strive to make the world a better place. You can find out about them and our partnerships in the Partners page.

3.4 Read articles

In order to gain extra information on how to act during a crisis, we have provided our website with different articles in many areas like First Aid, Events, Certificates, Volunteering. Every article is being shown in the Articles page, and has a short description in order to find out what it is about.

In order to read the full article, you need to press the button "Continue Reading" which appears for every article under its short description.

3.5 Contact Us

We have provided a Contact page where you can find out more information about Crisis Center, who we are and what is our motivation and our mission.

Nevertheless, if you have any feedback or question for us, there is a Contact form that you can submit which will be sent to us via email.

In order to submit it, you have to complete all the required fields and press the button "Submit".

4. MVC Model

Model-View-Controller (MVC) is an arhitectural pattern that separates an application into three main logical components: the model, the view and the controller. Each of these components are built to handle specific development aspects of an application.

Advantages include:

  • easier maintenance by keeping the data-manipulating logic separate
  • modularity, decoupled code (fewer dependencies)
  • modification not affecting the entire model
  • scalability
  • extensiblity

4.1 Model

Responsible for the business domain state knowledge. The model in our application wil consist of the database entities created.

4.2 View

Responsible for a presentation view of the business domain. It will be represented by html files for every page in the website.

4.3 Controller

Responsible for controlling the flow and state of the user input. Our application implements php scripts, using the models and generating the views.

6. Functionalities

6.1 Emergency Map

A map will be provided for both registered users and guests. The map will be shown in the home page will be created using Google Crisis Map API.

6.2 New Alert

The users that represent an authority will have the option to add a new alert for all of the users and guests. For this, they will have to fill in a form with some important details. For example, an important detail would be the location of the alert; this will be given with latitude and longitude coordinates.

6.3 New Missing Person

For a specific alert that is still open, users may add a missing person, even if it is an authority type of user or a normal user. This would be provided via a form, for important details about de person, like name, age or details regarding last time he/she was seen. A missing person can be added without mentioning a specific alert.

6.4 Find A Person

Whenever a missing person is found, the website will send a notification to all the users to verify that he/she really is the person that the user is trying to find. If the answer to this is affirmative, then the "Missing Person" alert will close and will be logically deleted from the database, not phisically, because we want a full history of alerts for every user.

6.5 Contact Us

Anyone(logged/unlogged users) can use the form in Contact page to communicate with the team behind this project, either to give feedback or to ask questions regarding this project.

6.6 Login

The Login process will be done via a form with an email and password. They will be verified with the database, with the important mention that the password will be sent already encrypted to the server.

6.7 Register

This process will also be done via a form, asking for an email, password, name, type of user (citizen or authority), country, city, ZIP Code and phone number. After these fields are completed, the server will check if the email does not already exist in the databse. If the answer is negative, the data will be stored in a database, along with the password being encrypted.

6.8 Emergency Manager

This feature is referring to the possibility to close some emergencies when they're outdated or "Find Person" requests. As in "Find Person", the deletion will be a logical one, not a phisical one.

7. External APIs

7.1 Google Crisis Map API

Google Crisis Map API is the main soul of the news page and maily to the whole application. This will help to ilustrate layers related to different scenarios of emergency. We will have a local hosted server meant to distribute this API, one which will be customized with our own information. By our own information, we also reffer to custom formatted alerts obtained from other API's and news sources. At this stage we will implement a checker not to have multiple alerts reffering to the same location. This will enhance the user exeperience and the find a person use case. Furthermore, we will make use of fusion tables to keep the data formated ready to deliver to the View part, namely the google Crisis map API visualizer.

7.2 Google Maps

This will be intensively used in the context of identifying the longitutde and latitude (a canonical representation of a location) of a written address. This will also be useful to translate fixed coordinates to a human readable address. Also we will also integrate the visual google maps needed to identify a fixed location by clicking on the map. Also at this point of geolocation, we will make use of geolocation sevices to detect ones position.

7.3 Facebook API

Facebook API will be used to integrate a login system into our page. But we will use it more extensevly, meaning that we will gather as much information as we can to build a exhaustive detaliation of a missig person, or a specific event (of course respecting the privacy consent).

7.4 Twitter API

Twitter API will be used as the other social network related API, meaning that it will help the loging procedure, and also it will help tweeting to spread awareness. (als in the limit of privacy limits) This will also be available for the article section in case there are articles that can be easily found through Twitter related to the problem (this will most probably be done in a latter version).

7.4 Person Finder API

This API will keep track of the person finder component. This will gather data from the Google hosted API in order to deliver a resonable interface to the users. To simplify the process we will try to combine the APIs and keep the raw data in a responsible format.

7.6 Information gathering APIs

First of all we will consider using the National Flood Data API, a system used for monitorizing the issues relatd to flooding. This will also be figured in the google crisis map API by some layering means.
Secondly, Foreca Weather Warnings Feed API will also be helpful to model some weather realted issues, which will ultimately be converted in custom alerts interpreted by our application (with respect to copyrights).
Finally, United Nations Humanitarian Response API is a huge uplift for our application, as it involves keeping track of emergencies world-wise, not only in USA as the Google Crisis Map does, or P2000 API does in Netherlands.

7.7 News API

As right now we don't need such a carefully selected media, we can just stick to the News API which provides information from several media sources. This will help us to extend our alerts and interpret some of this news into out application. As this will be kindly difficult for a plain server processing, the news will be formatted and accepted by the admin only if it fits the our requirements. By now there is only human interaction for approving the content, but in the future there will be smarter analysis of the content.

7.8 Notification

At this point we will be figuring out some means of notification throughout diferent browsers, meaning that we need to find a way to alert the user in case out geolocation features togheter with server-side processing report the need of a notification. For this we will get data related to different browsers/operating systems, and at the point we will be able to develop a stable version, we will kick a version implying this feature.

8. Technologies

8.1 PHP backend

The backend will be implemented in PHP, as this will be the most accessible and responsible for this type of application. Furthermore, the XAMPP stack will be most accessible through this type of backend. Our own REST API will also be supported in PHP as to keep a homogeneous environment of working.

8.2 MariaDB

This type of database is most suitable for the application as it is also resonable with this type of stack, namely XAMPP. This SQL-based free database is powerful enough to deliver a fast and consistent bulk of information. For developing purposes and at some extend for the efficient part, we will deplo this database on AWS.

8.3 HMTL CSS Javascript

This is the main soul of our front-end part. This is understandable as we want to have full control of our front-end, thus using a raw version of design modelling. This part is mostly complete, but it will also include some AJAX as we want to keep our communication with our server smooth and clean.

8.4 Version control - Git

This is a common choice for this type of applications, thus we will try to make use of it fully. There is not much to say related to this choice or why, aside the fact that we will be restricted to keep our repository public, watsoever.

8.5 Server - Apache

As the previous choice of technologies point towards XAMPP, endly the Apache is a natural choice.

8.6 Microservices - CAP

We will make use of several microservices, from which the the most important is CAP, the common alerting protocol, one used, as the name stands, for alerting a user in case of an emercency. We are still keeping under thinking this part, as the integration must be responsible.
